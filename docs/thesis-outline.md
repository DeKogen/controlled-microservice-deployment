# Структура дипломной работы

Документ можно использовать как основу для текста диплома. Он описывает логику изложения проекта от постановки задачи до оценки результата.

## 1. Введение

Содержание раздела:

- актуальность микросервисной архитектуры;
- проблема согласованного обновления API;
- роль Kubernetes и сервисной сети;
- цель работы;
- объект и предмет исследования;
- краткое описание разработанной системы.

Возможная формулировка цели:

> Цель работы - разработать систему контролируемого развертывания микросервисов, которая учитывает совместимость API-контрактов при выборе маршрутов между версиями сервисов.

## 2. Анализ предметной области

Что раскрыть:

- микросервисная архитектура и межсервисные зависимости;
- версионирование API;
- проблемы backward compatibility;
- Kubernetes Deployment и rolling update;
- сервисная сеть Istio;
- OpenAPI как формат описания HTTP API;
- недостатки ручной настройки маршрутов.

Ключевой вывод раздела:

> Стандартные инструменты инфраструктуры управляют состоянием сервисов и трафиком, но не принимают решения на основе совместимости API.

## 3. Постановка задачи

Задачи проекта:

1. Описать API-зависимость между сервисами на уровне кода.
2. Сгенерировать метаданные о потребляемых и предоставляемых API.
3. Проверить совместимость API-контрактов.
4. Описать логический сервис и его версии через Kubernetes custom resource.
5. Сформировать маршруты между совместимыми версиями.
6. Показать работу системы на демонстрационном приложении.

## 4. Проектирование системы

Раздел должен включать:

- общую архитектуру;
- состав модулей;
- схему API metadata flow;
- схему deployment flow;
- схему routing flow;
- структуру `VirtualDeployment`;
- взаимодействие с Kubernetes и Istio.

Опорные документы:

- [architecture.md](architecture.md)
- [diagrams.md](diagrams.md)
- [algorithm.md](algorithm.md)

## 5. Реализация

Подразделы:

### 5.1 API contract module

Описание `demo-shop-api`, DTO и интерфейсов.

### 5.2 Spring Boot integration

Описание remote client, metadata endpoints и генерации OpenAPI.

### 5.3 Orchestrator service

Описание наблюдения за ресурсами, проверки совместимости, генерации маршрутов и работы с Kubernetes/Istio API.

### 5.4 Demo services

Описание `demo-shop-cart` и `demo-shop-product`.

### 5.5 Kubernetes manifests

Описание CRD, namespace, gateway, services, deployments и routing resources.

## 9. Заключение

Заключение должно ответить на вопросы:

- достигнута ли цель работы;
- какие компоненты разработаны;
- как система повышает контролируемость развертывания;
- как проект можно развивать дальше.

Возможные направления развития:

- оформить orchestrator как полноценный Kubernetes operator;
- добавить более строгую RBAC-модель;
- добавить dashboard для графа совместимости;
- расширить поддержку API protocols;
- добавить полноценные end-to-end tests в ephemeral cluster.
